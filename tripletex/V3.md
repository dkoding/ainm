# V3

## Goal

Define a V3 architecture that upgrades the current `v2` planner from local heuristic catalog retrieval to a retrieval-backed Gemini planner that can use Vertex RAG or another pluggable retrieval backend.

V3 is not a ground-up rewrite.

It is a planner-context upgrade on top of the existing stable execution stack:

`/solve -> planner -> bridge JSON -> router -> wrapper/raw executor -> Tripletex proxy`

## Non-Negotiable Constraints

V3 must keep the existing repository and competition invariants:

- one HTTPS `POST /solve`
- request accepts `prompt`, `files[]`, `tripletex_credentials.base_url`, and `tripletex_credentials.session_token`
- response is HTTP `200` with exactly `{"status":"completed"}`
- all Tripletex calls use the provided proxy with Basic Auth `0:session_token`
- attachments and multilingual prompts remain first-class inputs
- execution is judged on final Tripletex state, then efficiency
- credentials stay outside the LLM planning payload
- architecture remains `HUMAN -> LLM -> JSON -> FLOWS/COMMANDS -> Tripletex`
- exact raw `operationId` fallback remains mandatory

## What `v2` Already Gets Right

The current implementation already has the correct high-level boundary:

- Gemini plans into a bridge JSON document rather than sending HTTP itself
- the router validates the bridge before execution
- the executor builds the actual HTTP request from catalog metadata
- the transport injects `base_url` and `session_token`
- attachments are preprocessed into evidence and then enriched through a separate attachment-fact extraction pass
- wrapper flows, friendly commands, and raw `operationId` fallback already exist

This means V3 should preserve:

- `app/solver.py`
- `app/router/`
- `app/raw/`
- `app/wrapper/`
- bridge validation
- OpenAPI-driven request-body validation

The unstable part is not execution.

The unstable part is planner context selection.

## What V3 Changes

V3 changes how Gemini is given Tripletex API knowledge.

`v2` approach:

- build a local context slice from generated catalogs
- send a broad allow-list plus top-ranked raw operations to Gemini
- rely on prompt rules and local validation

V3 approach:

- keep local catalogs as the authority
- add a retrieval layer that selects a compact candidate set from operation-, command-, and flow-level documents
- send Gemini only the smallest correct candidate context
- optionally tighten model control further with candidate-bounded structured output or function-like selection

The retrieval backend should be pluggable:

- local heuristic retrieval
- local hybrid retrieval
- Vertex RAG / Vertex Vector Search

The planner contract stays the same.

## Core V3 Rule

Gemini must not produce raw HTTP as the primary planning primitive.

Gemini should produce one of:

- a business flow selection
- a friendly command selection
- an exact raw `operationId` selection

plus the arguments needed for that selection.

The executor then derives:

- HTTP method
- path
- query parameters
- JSON body
- multipart body
- auth

This preserves determinism and keeps credentials out of the model layer.

## Target Architecture

### 1. Static authority layer

Local generated artifacts remain the source of truth:

- `app/generated/operation_catalog.json`
- `app/generated/command_catalog.json`
- `app/generated/flow_catalog.json`
- `app/generated/conformance_policies.json`
- `docs/openapi.json`

These local artifacts continue to drive:

- validation
- request shaping
- body-field legality
- final execution

Vertex retrieval is not the authority.

It is only a context-retrieval accelerator.

### 2. Retrieval corpus layer

Build retrieval documents at operation level, not whole-spec level.

Each raw-operation document should contain:

- `operationId`
- HTTP method
- path
- domain
- subdomain
- purpose
- semantic aliases
- technical flow family
- conformance policy key
- required inputs
- optional inputs
- path params
- query params
- request body kind
- writable body fields
- key examples or anti-patterns

Each wrapper command document should contain:

- command name
- mapped `operationId`
- legal inputs
- required inputs
- selector and payload semantics
- conformance notes

Each business-flow document should contain:

- flow name
- command sequence
- legal inputs
- prerequisite rules
- preferred use cases
- anti-use cases

### 3. Retrieval metadata

Metadata should support deterministic filtering and reranking.

Recommended metadata keys:

- `doc_type`
- `operation_id`
- `command_name`
- `flow_name`
- `domain`
- `subdomain`
- `method`
- `technical_flow_family`
- `conformance_policy_key`
- `request_body_kind`
- `is_mutation`
- `is_attachment_related`
- `requires_date_window`

### 4. Request-time planning pipeline

At solve time, the pipeline should be:

1. receive `/solve`
2. build attachment evidence
3. enrich with typed attachment facts
4. derive deterministic prompt hints locally
5. query retrieval backend for candidate operations/commands/flows
6. rerank locally with exact-match logic
7. assemble a compact context pack
8. ask Gemini for strict bridge JSON
9. validate bridge locally
10. execute through router and executor

## Retrieval Strategy

V3 should not rely on vector similarity alone.

OpenAPI retrieval has three separate matching modes:

- exact symbolic match
  - `operationId`
  - command names
  - field names
  - path fragments
- business-semantic match
  - “register payment”
  - “credit note”
  - “bookkeep attached supplier invoice”
- anti-confusion match
  - avoid `create` when the task is `reverse`
  - avoid direct delete when the correct action is `credit note` or `reverse`

Therefore V3 retrieval should be hybrid:

- metadata filtering
- semantic retrieval
- local lexical/exact reranking

This is the biggest difference from the current `v2` token-scoring catalog slice.

## Prompt Footprint Change

`v2` still sends a large amount of contract data every call:

- all legal flow names
- all legal command names
- all legal raw operationIds
- top-ranked raw operations

V3 should reduce that to:

- only the candidate flows
- only the candidate commands
- only the candidate raw operations
- only the relevant selector and payload families
- only the relevant conformance notes

This should lower:

- prompt size
- latency
- planner confusion
- repair frequency

## Attachment Handling in V3

V3 should keep the existing two-stage attachment path:

- local evidence extraction
- separate attachment-fact extraction

Static API retrieval and request-time attachment understanding solve different problems.

The Vertex retrieval layer should index static API/planner knowledge only.

It should not try to store per-request user attachments.

Per-request attachment facts should continue to enter Gemini through the request payload.

## Recommended Model Control in V3

The current `v2` model control is:

- JSON-only prompt rules
- `responseMimeType=application/json`
- local validation
- bounded repair

V3 should consider a tighter candidate-bounded output contract:

- Gemini still returns bridge JSON
- but selected flow/command/operation choices must come only from retrieved candidates
- later, candidate operations may be exposed as structured callable choices if that improves reliability

Even if function calling is introduced later, the model still should not directly perform HTTP.

## Gap Analysis From `v2`

| Area | `v2` status | V3 target | Delta |
| --- | --- | --- | --- |
| Retrieval backend | local heuristic token scoring | pluggable hybrid retrieval with Vertex option | medium |
| Retrieval corpus | generated catalogs only, in-process | retrieval documents per operation/command/flow | medium |
| Prompt footprint | broad allow-lists plus ranked raw ops | compact top-k candidate context only | high |
| Planner control | JSON prompting plus validator | candidate-bounded structured planning | medium |
| Authority model | local catalogs are authority | keep local catalogs authoritative | no change |
| HTTP execution | deterministic executor and transport | preserve as-is | no change |
| Credentials boundary | already separated from LLM | preserve as-is | no change |
| Attachments | evidence + typed extraction exist | preserve and feed into planner | low |
| Repair path | planner repair and execution-error repair | preserve initially, reduce over time | low |
| Backend dependency | local only | retrieval backend abstraction with local fallback | medium |

## Migration Plan

### Phase 1. Retrieval abstraction

Add a retrieval interface in front of the current `ContextCatalog`.

The interface should return:

- candidate flows
- candidate commands
- candidate raw operations
- supporting selector families
- supporting payload families
- supporting policy notes

This phase should keep the current local retrieval as the default implementation.

### Phase 2. Retrieval document export

Export operation-, command-, and flow-level retrieval documents from the existing generated catalogs.

Do not export the whole `openapi.json` as one document.

Use small documents with stable IDs.

### Phase 3. Hybrid local reranking

Before any Vertex dependency, improve local reranking to combine:

- exact operation name hits
- path/method hints
- semantic alias hits
- negative weighting for conflicting action families

This gives a stronger baseline and a fallback when Vertex is unavailable.

### Phase 4. Vertex retrieval backend

Implement a Vertex-backed retrieval provider.

Requirements:

- retrieval corpus built from exported documents
- metadata filters on domain/method/doc type when hints exist
- fallback to local retrieval on backend failure
- no execution dependency on Vertex retrieval correctness alone

### Phase 5. Compact prompt context

Once retrieval is stable, shrink the planner prompt:

- remove global allow-lists from the default prompt path
- send only retrieved candidates plus a smaller authority summary
- retain local validation against the full catalog after generation

### Phase 6. Candidate-bounded planner control

Tighten planner output so the model can only choose:

- one of the retrieved business flows
- one of the retrieved commands
- one of the retrieved raw operations

Local validation still remains mandatory.

## Explicit Rejections

V3 should explicitly reject these designs:

- full `openapi.json` attached to every Gemini request
- Gemini directly receives `session_token`
- Gemini directly performs HTTP or chooses arbitrary URLs without `operationId` validation
- Vertex retrieval used as the source of truth for legality
- replacing the current deterministic executor with free-form model-driven request assembly

## Why Vertex Retrieval Is Optional, Not Mandatory

The static API corpus is small enough that local retrieval remains viable.

The local corpus is roughly:

- `546` paths
- about `800` operations

So Vertex retrieval should be treated as an optimization and context-quality upgrade, not as a prerequisite for V3 correctness.

That implies:

- V3 must work with local retrieval only
- Vertex retrieval should improve candidate quality and prompt size
- fallback to local retrieval should remain available in production

## Success Criteria

V3 is successful if it improves planner quality without disturbing the stable execution boundary.

Concrete success criteria:

- fewer planner repairs
- smaller prompt context per request
- fewer invalid flow/command/operation selections
- fewer execution-time `400` / `409` / `422` retries caused by planner misbinding
- no leakage of credentials into the LLM layer
- no regression in raw `operationId` fallback coverage
- no regression in attachment-aware planning

## Recommended Default Direction

The best V3 direction is:

- keep the current `v2` executor and validator stack
- add a pluggable retrieval layer
- start with stronger local hybrid retrieval
- add Vertex retrieval as an optional backend
- continue planning in terms of flows, commands, and exact raw `operationId`s

That produces a V3 that is tighter, smaller, and easier for Gemini to use, without giving up the strongest part of the current system: deterministic execution.

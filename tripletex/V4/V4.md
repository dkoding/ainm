# V4

## Goal

Build a benchmark-oriented Tripletex solver that treats the competition as a finite closed-world task set, not as an open-ended accounting assistant.

V4 should keep the useful execution scaffold from the current runtime:

- `/solve` HTTP entrypoint
- proxy-only Tripletex transport
- OpenAPI-driven request shaping and validation
- wrapper/flow execution
- attachment preprocessing
- deploy, Docker, and Cloud Run tooling

V4 should replace the unstable planner-centric middle of the system.

The core design shift is:

- `v1-v3`: general LLM planner tries to invent a valid bridge JSON for any accounting request
- `v4`: deterministic benchmark solver classifies the request into one of the known Tripletex task families, extracts only the slots needed for that family, and then runs a fixed executor for that family

## Competition Facts V4 Must Exploit

- There are `30` task types.
- Each task type has `56` variants.
- Those variants come from `7` languages and `8` data sets.
- Some tasks include PDF or image attachments.
- Every submission starts from a fresh Tripletex account.
- Scoring is field-by-field correctness plus efficiency.

These facts change the optimal architecture.

This is not an open-domain agent problem.

It is a bounded multilingual benchmark with:

- known task families
- known end states
- known API surface
- known efficiency incentives

The winning architecture should therefore be:

- task-family classification first
- slot extraction second
- deterministic executor third
- narrow verification last

not:

- giant prompt
- giant retrieved catalog
- free-form bridge JSON planning
- router cleanup after the fact

## Why The Current Approach Scores Poorly

The log history shows repeated failure before we even reach correct Tripletex execution.

Root failure buckets observed in production:

- LLM contract and schema instability
- planner routing and catalog mismatch
- type coercion and input-shape failures
- over-blocking and fake multi-turn behavior
- Gemini context overflow and timeout pressure
- real Tripletex validation and restricted-endpoint failures

The highest-frequency failures are not business-logic failures inside Tripletex.

They are architecture failures in the planner layer.

### Root issue 1: we ask Gemini to generate too much structure

The current runtime asks Gemini to emit a large bridge object that includes:

- normalized language section
- semantic understanding section
- source section
- rich data section
- flat bridge section
- step plan
- validation section
- completion section

That is too much surface area for a benchmark where the actual decision is usually:

- which of 30 families is this
- what slots are present
- which fixed executor should run

This is why the logs contain:

- invalid JSON
- missing sections
- wrong section types
- unknown commands
- wrong step shapes

### Root issue 2: we are solving a closed-world benchmark like an open-world assistant

The current planner behaves as if it must reason over arbitrary accounting tasks and the whole API at once.

The competition docs say otherwise.

The search space is finite enough that we should explicitly model:

- task family
- required slots
- allowed execution paths
- completion criteria

### Root issue 3: we let the model choose raw operations too early

Free-form planning allows the model to:

- invent unsupported command names
- choose the wrong abstraction level
- choose direct mutations when a business flow exists
- emit illegal input shapes

V4 should invert control:

- the runtime chooses the executor family
- the model only fills slots inside that family schema

### Root issue 4: we waste model context on API retrieval instead of task resolution

The recent `Vertex AI returned HTTP 400` failures proved the prompt package exceeded Gemini's `1048576` token input limit.

This happened because we kept expanding planner context rather than reducing the problem.

V4 should never send broad command catalogs for ordinary tasks.

It should send:

- the user prompt
- attachment facts
- fresh-account assumptions
- a short list of candidate task families
- the one family schema the model must fill

### Root issue 5: we block too often instead of using benchmark assumptions

Fresh-account submissions make many clarifying searches unnecessary.

The current planner still blocks on:

- “subsequent turn” logic
- missing data that should be inferred from the task family
- missing identifiers that should be resolved within the executor

V4 should only block when the end state is truly underdetermined.

## V4 Design Principles

### 1. Task family is the primary planning unit

The main internal object should be `task_family`, not `flow`, `command`, or `raw_operation`.

Each family should define:

- canonical family name
- supported languages and phrasing patterns
- attachment requirement
- required slots
- optional slots
- deterministic executor
- allowed fallback behavior
- verification rules
- efficiency budget
- block conditions

### 2. The LLM should extract slots, not build execution graphs

Gemini should not emit a full bridge plan in the normal V4 path.

Gemini should emit a compact structured object such as:

- `taskFamily`
- `confidence`
- `language`
- `attachmentFacts`
- `slots`
- `missingRequiredSlots`
- `ambiguities`

The router and executors should then turn that into concrete wrapper/raw calls.

### 3. Deterministic local code owns routing after classification

Once the task family is known, local code should decide:

- which flow or command path to use
- which selectors to resolve
- which objects to create
- which verification reads to perform

This removes:

- invented command names
- invented step graphs
- malformed bridge sections
- most abstraction-level confusion

### 4. Fresh-account assumptions must be encoded directly

V4 should assume a clean company unless the task itself creates dependencies first.

That means family executors can often:

- create directly
- search narrowly only when the task implies an existing record
- skip broad exploratory scans

### 5. Attachments stay first-class

Attachment tasks are core benchmark tasks.

V4 must keep and harden:

- text extraction
- PDF/image multimodal evidence
- attachment fact extraction
- provenance tracking

But attachment data should feed a family-specific slot extractor, not a general planner.

### 6. OpenAPI remains the execution authority

V4 does not replace the current OpenAPI-driven transport and validation layer.

It keeps:

- request typing from `docs/openapi.json`
- schema guardrails
- body-field legality checks
- raw `operationId` coverage

The redesign is above that layer, not below it.

## Core V4 Pipeline

### Stage 1. Request normalization

Normalize incoming `/solve` into:

- prompt text
- language hints
- file inventory
- current date
- timezone
- request metadata

Output should be a small typed object with no execution decisions yet.

### Stage 2. Attachment evidence and facts

Reuse the current attachment pipeline to produce:

- extracted text
- OCR text
- multimodal Gemini facts when needed
- provenance and confidence

Attachment output should be normalized into slot-like evidence, not planner prose.

### Stage 3. Deterministic task-family shortlist

Before any LLM call, local code should produce a shortlist of likely task families from:

- lexical cues
- amount/date/entity patterns
- attachment presence
- language-independent slot signals
- previously known benchmark categories

The shortlist should usually be very small.

Target:

- `1-3` likely families, not `40-120` candidate operations

### Stage 4. Family classification

Gemini should be asked only to choose among the shortlisted families when local heuristics are not already decisive.

If local confidence is high enough, the LLM can be skipped.

If Gemini is used, the output should be a tiny JSON object, not the full bridge contract.

### Stage 5. Family-specific slot extraction

Once the family is chosen, run a second bounded extraction step if needed.

This extraction should be family-specific.

Examples:

- `employee.create_basic` family schema
- `invoice.register_payment` family schema
- `supplier_invoice.import_from_attachment` family schema
- `travel_expense.create_with_rows` family schema

Each schema should describe exactly:

- required slots
- optional slots
- slot types
- allowed derived values

### Stage 6. Deterministic family executor

Each family executor should:

- resolve selectors
- create prerequisites
- call wrapper flows or commands
- fall back to raw operations only where the family explicitly allows it
- perform bounded verification

The executor should own the control flow.

The model should not.

### Stage 7. Verification and completion

Verification should be family-specific and minimal.

Examples:

- for a create task: verify the created object exists with the expected key fields
- for a payment task: verify invoice state or payment linkage
- for a correction task: verify the reversal or delete effect only

No broad “re-read everything” behavior.

## Task Family Manifest Design

V4 should define a manifest for all `30` benchmark families.

Each manifest entry should include:

- `familyName`
- `category`
- `description`
- `requiresAttachment`
- `canUseAttachment`
- `languages`
- `canonicalExamples`
- `negativeExamples`
- `requiredSlots`
- `optionalSlots`
- `derivedSlots`
- `executorName`
- `preferredFlows`
- `allowedCommands`
- `allowedRawOperations`
- `verificationSpec`
- `blockSpec`
- `efficiencyNotes`

This manifest becomes the real planning authority.

Not a free-form prompt.

## Slot Extraction Strategy

Slots should be typed and family-specific.

Examples of slot groups:

- person name parts
- email
- organization number
- invoice number
- amount
- VAT type
- account number
- department name
- project name
- travel date window
- attachment reference

For each slot, the system should track:

- value
- provenance
- confidence
- explicit versus derived

But the output must stay compact enough to be executed deterministically.

## Multilingual Strategy

V4 should still support all competition languages.

The difference is where multilingual complexity lives.

It should live in:

- classification
- slot extraction
- attachment fact extraction

It should not live in:

- free-form execution planning
- open-ended bridge composition

The benchmark facts make this practical because the underlying task families are fixed even though phrasing varies.

## Attachment Strategy

Attachments must remain on the core path.

V4 should keep:

- attachment metadata
- text extraction
- OCR/multimodal extraction
- JSON salvage for noisy model output

V4 should add:

- family-specific attachment slot mapping
- explicit attachment-required family manifests
- stronger distinction between:
  - attachment failed to parse
  - attachment parsed but task still blocked
  - attachment parsed and executor can proceed

## Execution Strategy

Executors should be hand-authored for the benchmark families that drive score.

This is the right tradeoff because:

- there are only `30` task types
- the account is fresh each run
- efficiency matters
- action endpoints and wrapper flows can be chosen exactly

The main optimization goal is not genericity.

It is repeatable benchmark performance.

## OpenAPI and Wrapper Reuse

V4 should reuse the existing stable execution boundary:

- `openapi_catalog`
- `input_coercion`
- `openapi_schema_guard`
- raw transport
- wrapper commands
- wrapper flows
- router execution

But V4 should stop exposing these layers directly to the LLM as the primary planning space.

The family executor should choose among them locally.

## Evaluation and Telemetry

V4 needs explicit benchmark telemetry, not just raw error logs.

Per request, record:

- classified family
- shortlist families
- whether classification used Gemini
- extracted slot set
- blocked versus executed
- wrapper flow count
- command count
- raw operation count
- 4xx count
- total latency
- verification latency
- final outcome bucket

This is necessary to improve score systematically instead of reading scattered Cloud Run logs after the fact.

## Rollout Strategy

V4 should be built in phases.

### Phase 1. Scaffold fork and planning

- copy the current execution scaffold into `V4/`
- isolate V4 deploy config from the live service
- define the family-manifest format
- define the benchmark telemetry format

### Phase 2. Family registry and classifier

- enumerate the `30` benchmark families
- write the initial manifest
- build deterministic shortlist logic
- build compact Gemini family classification

### Phase 3. Family slot schemas

- define per-family schemas
- implement family-specific extraction prompts
- harden attachment-to-slot extraction

### Phase 4. Deterministic executors

- implement the first high-impact families
- route them through current wrapper/raw code
- verify final state with minimal reads

### Phase 5. Replace bridge-first planning on the hot path

- make benchmark-family execution the default path
- keep the old bridge planner only as a fallback research path, if at all

### Phase 6. Score-driven iteration

- run repeated task evaluations
- inspect failures by family, not by individual error string
- tighten the specific family manifest, extractor, and executor

## Initial V4 Deliverables

The first concrete V4 work should produce:

- isolated `V4/` runtime scaffold
- benchmark family manifest format
- list of all target task families
- compact family-classification interface
- compact family-slot extraction interface
- deterministic executor registry skeleton
- per-family verification skeleton

## Success Criteria

V4 is successful only if it changes the failure profile.

The target is to eliminate entire failure classes, not slightly reduce them.

The most important classes to remove are:

- invalid planner JSON
- planner bridge schema mismatches
- invented command names
- direct-mutation routing mistakes
- huge planner-context token overflows
- fake multi-turn blocking on one-shot benchmark tasks

If those remain common, V4 has not actually changed the architecture enough.
